# x402 Payments

Elytro speaks the x402 "exact" scheme over HTTP so agents can pay metered APIs directly from their smart accounts without touching the owner EOA. The CLI prefers ERC-7710 delegations, but it can also synthesize EIP-3009 (`transferWithAuthorization`) signatures for USDC-style paywalls that only offer the legacy flow.

## Prerequisites

1. **Deployed smart account** -- The account must be deployed on-chain (not just a counterfactual address). For EIP-3009 payments, the USDC contract calls `isValidSignature` (ERC-1271) on the account address; this reverts if there is no bytecode. Run `elytro account info` to check deployment status. If not deployed, run `elytro account activate` first.

2. **Delegation (ERC-7710 path)** -- An ERC-7710 proof that authorizes a facilitator to move a specific token/amount on your behalf. If the server only offers EIP-3009, skip this step; the CLI will auto-sign.

   ```bash
   elytro delegation add \
     --account agent-primary \
     --manager 0xDelegationManager \
     --token 0xUSDC \
     --payee 0xMerchant \
     --amount 1000000 \
     --permission 0xabc123... \
     --verify
   ```

   The `--verify` flag simulates the delegation on-chain before storing, catching invalid permission contexts early.

3. **Unlocked wallet & sufficient balance** -- same requirements as other Elytro commands. For EIP-3009, confirm token balance with `elytro query balance --token <asset>`.

## Delegation Lifecycle

### Import and verify

Store a delegation provided by the API server, then confirm it works:

```bash
elytro delegation add --manager 0x... --token 0x... --payee 0x... --amount 1000000 --permission 0x...
elytro delegation verify <id>
```

Verification performs three checks: local expiry, token balance sufficiency, and on-chain simulation of `redeemDelegations`. Status can be `valid`, `expired`, `insufficient_balance`, or `invalid_onchain`.

### Batch sync

Validate all stored delegations at once and optionally remove expired ones:

```bash
elytro delegation sync              # verify all, report statuses
elytro delegation sync --prune      # same, but auto-remove expired
```

### Renew

When a delegation is approaching expiry, renew it with extended time. If the on-chain delegation was recreated, pass the new permission context:

```bash
elytro delegation renew <id> \
  --expires-at 2026-04-01T00:00:00Z \
  --permission 0xnewContext... \
  --remove-old
```

This creates a new local delegation with the same parameters but updated expiry, and optionally removes the old record.

### Revoke

To revoke a delegation on-chain and remove it locally:

```bash
elytro delegation revoke <id> --calldata 0x...
```

The `--calldata` is the DelegationManager-specific revocation function call. Without it, only the local record is removed. Use `--keep-local` to retain the local record after on-chain revocation (useful for debugging).

### Local-only remove

To remove a delegation record without revoking on-chain:

```bash
elytro delegation remove <id>
```

## Making a Paid Request

```bash
elytro request https://x402.example.com/paywall \
  --method POST \
  --json '{"topic":"defi"}'
```

Flow:

1. The CLI sends your request normally.
2. If the server responds `402 Payment Required`, Elytro decodes the `PAYMENT-REQUIRED` header, selects a matching payment option for your chain (preferring ERC-7710 over EIP-3009), and builds a signed `PaymentPayload`.
3. The CLI replays the HTTP call with the `PAYMENT-SIGNATURE` header. On success, it reports: resource details, token and amount sent, delegation manager or authorization method used, settlement transaction hash and network, and the original response body.

### Dry Run

```bash
elytro request --dry-run https://x402.example.com/paywall
```

Prints the paywall terms without sending funds. Always dry-run first so the user can review the price.

### Verbose Mode

```bash
elytro request --verbose https://x402.example.com/paywall
```

Logs full request/response headers and the EIP-3009/ERC-7710 signing flow. Use `--verbose` when diagnosing payment failures to see the exact facilitator response and signature parameters.

### Result Types

The `request` command returns one of four result types:

- **`preview`** (dry-run): payment requirement details, no funds moved.
- **`paid`**: payment succeeded. Includes `responseBody` (parsed JSON), `payment.settlement`, and `payment.authorization`.
- **`payment_failed`**: payment was attempted but rejected. The error output includes `facilitatorResponse` (the full parsed response from the facilitator/server) and `suggestion` (an actionable next command based on the failure pattern). Read both fields to diagnose.
- **`plain`**: the server did not return 402; no payment was needed.

## Limitations

- ERC-7710 is the preferred path. When it's missing, Elytro falls back to EIP-3009 on `eip155:*` networks (requires `extra.name`/`extra.version` so the typed data domain matches the token). Permit2 remains unsupported.
- Some servers embed the `PaymentRequired` JSON in the response body instead of the `PAYMENT-REQUIRED` header. The CLI parses both, but header-compliant servers are more reliable.
- Delegations must be refreshed before expiration; expired proofs cause payments to fail. Use `delegation sync --prune` to clean up, then `delegation renew` or `delegation add` with fresh credentials.
- Extensions advertised in `PAYMENT-REQUIRED` are echoed verbatim but not interpreted client-side yet.
- Response bodies (`responseBody`, `facilitatorResponse`) are parsed as structured JSON when possible, falling back to raw strings for non-JSON responses. This avoids double-serialization in the output.

## Troubleshooting

| Error                                                   | Fix                                                                                                                                                                                                                                       |
| ------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `No stored delegation matches this payment requirement` | Run `delegation list` to check stored delegations. Ensure manager, token, payee, and amount all match the server's requirements. Add a matching delegation with `delegation add`.                                                         |
| `No payment option offered for network ...`             | The server doesn't support your chain. Switch to a supported chain or ask the provider.                                                                                                                                                   |
| `Delegation verification failed: expired`               | The delegation has expired. Run `delegation renew` with a new expiry and permission context, or `delegation add` a fresh one.                                                                                                             |
| `Delegation verification failed: insufficient_balance`  | The smart account lacks sufficient token balance for the delegation amount. Fund the account first.                                                                                                                                       |
| `Delegation verification failed: invalid_onchain`       | The permission context is not accepted by the DelegationManager contract. It may have been revoked or never created on-chain. Obtain a new delegation from the server.                                                                    |
| `Permission context expired`                            | Same as the expired case above. Renew or replace the delegation.                                                                                                                                                                          |
| `Delegation manager revert`                             | Verify the payee matches the requirement and that the delegation amount covers the invoice. Run `delegation verify <id>` for details.                                                                                                     |
| `payment_failed` with generic "Payment required"        | Facilitator returned a generic rejection. The account may not be deployed (ERC-1271 fails without bytecode). Run `account info` to check, then `account activate` if needed. Retry with `--verbose` to see the full facilitator response. |
| `payment_failed` with "invalid_signature"               | EIP-3009 signature rejected. Verify the account is deployed (`account info`) and the EIP-712 domain matches (`extra.name`/`extra.version` in the payment requirement).                                                                    |
| `payment_failed` with "insufficient_balance"            | Token balance too low for the payment amount. Check with `query balance --token <asset>` and fund the account.                                                                                                                            |
| `payment_failed` with "simulation_failed"               | On-chain simulation reverted. Likely causes: insufficient token balance, undeployed account, or invalid delegation. Run `account info` and `query balance --token <asset>`.                                                               |
| `payment_failed` with "expired" or "timeout"            | Payment authorization expired before settlement. Check system clock, retry with shorter latency, or renew delegation.                                                                                                                     |
