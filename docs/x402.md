# x402 Payments (Preview)

Elytro now speaks the x402 “exact” scheme over HTTP so agents can pay metered APIs directly from their smart accounts without touching the owner EOA. The CLI prefers ERC-7710 delegations, but it can also synthesize EIP-3009 (`transferWithAuthorization`) signatures for USDC-style paywalls that only offer the legacy flow.

## Prerequisites

1. **Delegation** — ERC-7710 proof that authorizes Elytro’s facilitator to move a specific token/amount on your behalf. (Skip this if the server only offers EIP-3009; in that case the CLI will self-sign an authorization for you.)

   ```bash
   elytro delegation create \
     --account agent-primary \
     --manager 0xDelegationManager \
     --token 0xUSDC \
     --amount 10 \
     --payee 0xMerchant \
     --expires-in 3600
   ```

   Reuse delegations for multiple sessions or revoke them anytime with `elytro delegation revoke`.

2. **Unlocked wallet & active account** — same requirements as other Elytro commands.

## Making a Paid Request

```bash
elytro request https://x402.example.com/paywall \
  --method POST \
  --json '{"topic":"defi"}'
```

Flow:

1. The CLI sends your request normally.
2. If the server responds `402 Payment Required`, Elytro decodes `PAYMENT-REQUIRED`, finds an ERC-7710 offer (or falls back to EIP-3009 on EVM networks), and builds a `PaymentPayload` signed via ERC-1271.
3. The CLI replays the HTTP call with `PAYMENT-SIGNATURE`. On success, it shows:
   - Resource details
   - Token + amount sent
   - Delegation manager used
   - Settlement transaction hash / network
   - Original response body

### Dry Run

```bash
elytro request --dry-run https://x402.example.com/paywall
```

Prints the paywall terms without sending funds.

## Limitations

- ERC-7710 is the preferred path. When it’s missing, Elytro falls back to EIP-3009 on `eip155:*` networks (requires `extra.name`/`extra.version` so the typed data domain matches the token). Permit2 remains unsupported.
- Some servers still embed the `PaymentRequired` JSON in the 402 response body instead of the `PAYMENT-REQUIRED` header. The CLI now falls back to parsing the body, but header-compliant transports are more reliable (extensions, settlement headers, etc.).
- Delegations must be refreshed before expiration; expired proofs cause payments to fail with `invalid_permission_context`.
- Extensions advertised in `PAYMENT-REQUIRED` are echoed verbatim but not interpreted client-side yet.

## Troubleshooting

| Error | Fix |
| ----- | --- |
| `No ERC-7710 payment option` | Ask the provider for smart-wallet support or fund an EOA for Permit2/EIP-3009. |
| `Permission context expired` | Create a new delegation with `elytro delegation create`. |
| `Delegation manager revert` | Verify the payee matches the requirement and that the delegation amount covers the invoice. |
